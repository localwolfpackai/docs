## Code Blocks

Every page on this site uses fenced code blocks. Here's the syntax reference I come back to.

### Basic Block

Add the language after the opening backticks for syntax highlighting. Optionally add a filename after the language.

```python agent.py
def run_agent(task: str) -> str:
    """Send a task to Claude Code and return the result."""
    result = subprocess.run(
        ["claude", "-p", task],
        capture_output=True, text=True
    )
    return result.stdout
```

## Claude Code

My primary coding agent. Here's how I actually use it.

### Quick Commands

The commands I run most:

```bash
# start a conversation
claude

# one-shot prompt (no interactive session)
claude -p "add error handling to src/api.ts"

# continue the last conversation
claude --continue

# resume a specific session
claude --resume
```

### CLAUDE.md Files

This is the most important config. A `CLAUDE.md` file in your project root gives the agent persistent context about your codebase — coding conventions, project structure, how to run tests, what to avoid.

```md CLAUDE.md
# Project: my-api

## Stack
- Node.js + TypeScript
- PostgreSQL with Drizzle ORM
- Vitest for testing

## Commands
- `npm run dev` — start dev server
- `npm test` — run test suite
- `npm run lint` — lint and format

## Rules
- Use snake_case for database columns
- All API responses use the { data, error } envelope
- Never commit .env files
```

<Tip>
  The agent reads `CLAUDE.md` at the start of every session. Keep it concise and up to date — this is the single highest-leverage thing you can do to improve agent output.
</Tip>

### Skills

Skills are reusable knowledge packages that extend what the agent knows. I use [skills.sh](https://skills.sh) to manage them.

```bash
# add a skill
npx skills add claude-code

# search for skills
npx skills search

# list installed skills
npx skills list
```

### Hooks

Hooks let you run shell commands automatically in response to agent actions — before/after tool calls, on session start, etc.

```json .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "echo 'Running bash command...'" }
        ]
      }
    ]
  }
}
```

### Agent Patterns I Use

**Bug fix flow:**
1. Paste the error or describe the bug
2. Let the agent read the relevant files and trace the issue
3. Review the proposed fix
4. Ask it to write a test that would have caught it

**Feature flow:**
1. Describe what I want at a high level
2. Ask the agent to plan before writing code
3. Review the plan, adjust if needed
4. Let it implement, then review the diff

**Refactor flow:**
1. Point the agent at the file or function
2. Describe what I want changed (rename, extract, restructure)
3. Ask it to check for breakages across the codebase

<Warning>
  Always review agent-generated code before pushing. Agents are fast but not infallible — especially with complex state management or security-sensitive code.
</Warning>

## Best Practices

- **Always specify the language** after opening backticks for syntax highlighting
- **Add a filename** after the language to show a label above the block
- **Use `<CodeGroup>`** when showing the same thing in multiple languages
- **Keep examples real** — use actual commands and patterns from your workflow, not hello-world samples
